# 公开题库质量审计

审计日期：2026-07-11

主要范围：`data/cases.json`；为判断线上发布结果的真实性与安全边界，同时只读检查了题库入库、结果记录与 `iframe` 预览链路。

本次只做审计，不修改题目数据。

## 结论先行

这套题库已经适合做“中文模型输出画廊”：题目短、首屏易懂、输出差异可见，创意写作与单文件前端也很适合并排展示。但它还不适合被称为严格、可排名的 benchmark。主要缺口不是题目数量，而是三件事：

1. **结果与题目版本尚未形成不可变绑定**。管理员改 Prompt、系统提示或输出类型后，旧结果可能失去可解释性。
2. **绝大多数评分点只是四个短标签**，没有硬性检查、权重、1/3/5 分锚点或失败门槛，同一输出很难被稳定复评。
3. **HTML 结果是主动内容**。本轮已禁止文本内容嗅探升级，补齐 sandbox、referrer policy 与 CSP，并把外部资源收窄到 Three.js 与字体所需的指定 CDN；但内联脚本仍可长时间占用 CPU/GPU，进程级隔离尚未完成。

因此，管理员手动跑一遍可以作为“本次精选样例”，但应叫**单次实测快照**，不能据此宣称模型排名或稳定能力。若未来要做排行榜，确定性题至少固定参数并自动验收；创意题应每模型至少 3 次采样，再报告通过率或评分分布。

### 本轮整改状态

- 已完成：显式 `outputType` 优先；文本结果不再按内容嗅探升级为可执行 HTML；官方结果改为管理员显式发布的不可变版本；普通运行历史与精选结果隔离。
- 已完成基础隔离：管理员发布结果和当前浏览器主动运行的交互预览统一使用 `sandbox="allow-scripts"`、`referrerpolicy="no-referrer"`，并注入禁止表单、父级导航、嵌套 frame 与 `connect-src` 的 CSP；图片/媒体只允许 data/blob，外部脚本只允许 jsDelivr、unpkg，字体只允许 Google Fonts 对应域名。匿名社区贡献和普通公开历史不执行 HTML，只展示转义后的源码。
- 仍需后续完成：将允许的 Three.js/字体版本固定并自托管，把交互预览升级为按需启动、可停止的独立渲染环境，对 CPU、GPU、内存和运行时长设限。

## P0：发布结果页之前必须修

### P0-1 显式 `outputType` 会被关键词推断覆盖（本轮已修复）

`data/cases.json` 中 `write-product-launch` 和 `write-long-context-brief` 都明确为 `text`，但题面分别含“网页/前端”和“Three.js/定价页”。`lib/db.js` 与 `public/app.js` 的推断逻辑在显式值不是 `html` 时仍继续扫关键词，导致这两题在 DB/API 中被判为 `html`。

直接后果不是显示标签错，而是运行时会追加“只输出完整 HTML 或 SVG”的系统提示，模型实际收到的任务已经改变。

**整改：** `outputType` 只要显式为 `text` 或 `html` 就必须拥有最高优先级；推断只用于字段缺失的 legacy 记录。发布快照必须保存最终有效输出类型与最终有效 system prompt 的哈希。

### P0-2 禁止把文本结果按内容嗅探后升级成可执行 HTML（本轮已修复）

贡献与历史详情目前使用“记录为 html **或内容看起来像 HTML**”决定是否放入带脚本的 `srcdoc iframe`。这会让一条本来属于文本题、仅仅包含 HTML 示例的模型回答变成主动内容。

**整改：** 是否可执行只能由经过审核且冻结在题目/发布快照里的 `outputType` 决定；不可信输出内容只能降级，不能升级。未知类型默认纯文本。源码查看始终转义。

### P0-3 官方结果必须是管理员发布的不可变快照，不能复用公共历史（本轮已修复）

普通运行历史允许访客写入，适合“我的运行记录”，不适合做官方结果来源。若首页直接取“最新历史”，访客运行、部分成功、改写 Prompt 或错误模型标签都会污染官方页面。

**整改：** 保持两套概念：

- `run_history`：普通运行日志，不代表官方结论；
- `published_runs`：只有管理员能创建的不可变快照；
- `featured_run`：每题显式指向一个已发布版本，切换指针而不是覆盖旧数据。

一次管理员操作可以“一键运行全部并形成候选”，但必须在所有预期模型完成、输出可打开、元数据齐全后再点“发布”。失败或少模型的候选不能自动顶掉当前精选版本。

### P0-4 主动 HTML 预览仍可联网和消耗浏览器资源（基础隔离已补，长期方案待完成）

审计初始状态已有 `sandbox="allow-scripts"` 且没有 `allow-same-origin`，这是正确的隔离基础：结果不能直接读父页面 DOM。但 `srcdoc` 当时仍能加载任意脚本、图片、字体并发起网络请求；恶意或失控输出还可运行死循环、创建大量 Canvas/WebGL 资源或做追踪请求，贡献/历史入口也没有统一的 referrer policy。本轮已统一渲染入口、禁止 `connect-src` 并把外部资源限制到题库所需 CDN；运行资源限制仍是后续工作。

**最低发布门槛：**

- 所有预览入口共用同一安全渲染函数，禁止漏配属性；
- 永远不要加入 `allow-same-origin`、`allow-top-navigation`、`allow-popups`、`allow-forms`；
- 给子文档加 CSP，默认 `connect-src 'none'`、限制 `img-src/font-src/script-src`；
- Three.js 等依赖固定版本并自托管或严格白名单，不允许模型自行选择任意 CDN；
- 默认先展示静态截图与源码，交互预览按需启动并可一键停止/销毁；
- 更稳妥的长期方案是无 Cookie 的独立渲染域，并对 CPU、内存、运行时长做隔离。

## P1：首批正式结果前应处理

### P1-1 清理“第 33 题”口径与测试数据

当前第 33 条来自数据库中的社区测试投稿，不在 `data/cases.json`，内容只有“写一句话”，没有可用区分度。它不应进入官方题库、首页完成率或管理员待跑清单。

**整改：** 默认题库只统计 `source=official` 且具备有效 rubric 的题；社区题进入独立审核流。下线该测试项，或至少从正式结果页过滤。

### P1-2 把评分点从“标签”升级为可执行 rubric

当前四项如“美观”“视觉”“可读”“风格统一”无法让两个评审给出接近分数。建议每题拆成两层：

```json
{
  "hardChecks": [
    { "id": "valid_json", "weight": 0, "fatal": true },
    { "id": "six_items", "weight": 0, "fatal": true }
  ],
  "dimensions": [
    {
      "id": "actionability",
      "label": "可操作性",
      "weight": 0.35,
      "anchors": {
        "1": "大多只说失败，没有下一步",
        "3": "多数给出动作，但不够具体",
        "5": "六种场景均给出与原因匹配的明确动作"
      }
    }
  ]
}
```

硬性格式失败应与审美低分分开；不能让一个无效 JSON 靠“语气友好”拿到高总分。主观题优先做匿名、随机左右顺序的成对比较，并记录评审者与评分版本。

### P1-3 为高价值题补自动验证器

建议先覆盖最容易验证的一批：

- JSON/格式：`write-micro-copy`、`write-instruction-follow`、`code-regex-extract`；
- 约束文本：字数、行数、禁词、必含元素、标题与小节结构；
- HTML 静态结构：完整文档、禁止外部资源、指定 DOM/ARIA 属性；
- HTML 行为：用固定浏览器与 Playwright 验证添加、删除、拖拽、主题记忆、键盘操作、碰撞、重开；
- 前端质量：固定桌面/移动 viewport，检查控制台错误、横向溢出、可聚焦元素和基础 axe 规则。

验证器本身也要版本化。发布页应区分“自动检查通过”和“人工评价”，不要混成一个模糊总分。

### P1-4 重做分类体系

`creative-writing` 目前同时承载文学、产品文案、摘要、算法教学、安全审查、SQL 与架构推理，分类名会误导用户，也无法按能力看结果。

建议至少拆成：

- `creative-writing`：寓言、独白、小说、诗歌、风格改写；
- `product-writing`：发布文案、微文案、会议纪要；
- `instruction-following`：精确格式、约束遵循；
- `coding`：算法解释、代码审查、结构化抽取、SQL；
- `reasoning`：架构权衡、日志排障；
- `frontend`：再以 tags 区分 layout / interaction / a11y / graphics / web-components。

### P1-5 固定运行清单，避免“同题不同考法”

正式快照至少要保存：

- case id、case revision、完整 system/user prompt、二者哈希；
- 运行时额外追加的 system prompt；
- provider、请求 model id、服务商实际返回 model id；
- temperature、max tokens、是否省略 temperature、是否触发自动重试及重试参数；
- start/end time、latency、usage、finish reason、截断状态；
- 应用版本、渲染器版本；前端题还要有浏览器版本、viewport、DPR、主题和网络策略。

当前前端对站点模型固定使用 temperature 0.7，文本/HTML token 上限也不同；空结果重试还会改变参数。若不记录这些信息，页面上的结果无法重现。

### P1-6 固定或移除外部依赖

`fe-threejs-orbit` 与 `fe-threejs-particles` 允许从 jsDelivr/unpkg 引入 Three.js，但没有固定版本。相同输出可能今天可运行、以后因 CDN、版本或网络策略失效。

**整改：** Prompt 指定精确版本与唯一允许 URL，最好由站点自托管；发布时保存依赖清单与完整性哈希。其他前端题明确“不得加载任何外部资源”，不要用“字体 CDN 以外的依赖”这类双重否定。

### P1-7 修复三道存在逻辑缺口的题

- `code-sql-optimize`：没有数据库方言、目标结果粒度、表结构、选择性和现有索引；原 SQL 对 coupons 的连接可能制造重复行。不同改写是否语义等价无法判断。
- `reason-debug-log`：日志时间线不自洽。`10:01:02` 已 200，随后同一秒声称等待 60000ms 超时，客户端又在 `10:01:03` 提前中止。好模型可能指出证据矛盾，差模型反而会顺着预设答案编故事。
- `fe-kanban-dnd`：Prompt 只要求“添加”，rubric 却评分“增删”，属于未告知要求。

这三题不宜直接跑成官方结果，应先补齐题面再创建 v2。

### P1-8 “长上下文”名不副实

`write-long-context-brief` 整个 Prompt 仅数百字符，本质是噪声纪要摘要，不是 long-context。它还包含相互冲突的预算表态，容易把“某人坚持/有人提议”误判为已决事项。

**整改：** 要么改名为“冲突纪要去噪”，并给出决策真值表；要么扩成 8k–20k 的合成材料，埋入可核验事实、干扰项和引用位置，真正测长上下文检索与压缩。

### P1-9 降低题库对单一品牌审美的过拟合

青绿、深色、禁紫、禁竖线本身不是问题，但在多个前端题重复出现，会让“记住乔木偏好”替代真实设计能力。7 题还直接复用同一品牌名，页面很容易长得相似。

**整改：** 保留约 1/3 作为乔木品牌题；其余使用不同受众、内容密度、主题与情境，并提供明确但互相不同的视觉约束。最终报告单独标记“品牌遵循”与“通用能力”，不要合并成单榜。

## 管理员跑一次，怎样才算可发布

“管理员跑一次作为结果”可以成立，前提是产品承诺是**展示当次实测**，不是统计排名。推荐流程：

1. 管理员选择固定模型集合，点击一次“运行全部”。
2. 系统创建 draft run，记录本次有效 Prompt、参数与模型身份。
3. 全部模型完成后自动跑格式/行为检查；部分失败时保留 draft，但不替换线上结果。
4. 管理员人工检查文本是否截断、HTML 是否正常、是否有不安全外链。
5. 点击“发布本次结果”；系统生成不可变版本，并把该题的 featured 指针切到新版本。
6. 后续重跑永远生成新版本；可回滚旧 featured，不覆盖历史。

页面必须明确显示“单次样例”“运行日期”“模型精确名称”“Prompt 版本”。若要说“谁更强”，再升级为：确定性题固定低温并重复验证；创意题每模型多次采样、盲评并展示方差。

### 发布门槛

- 预定模型数量全部成功；若允许缺席，页面明确标“未完成”，不能静默少一列；
- 输出非空、非截断、无仅推理内容；自动重试必须显示；
- 格式硬检查通过；
- HTML 无未知外链，固定 viewport 下无控制台致命错误；
- run snapshot 含完整元数据与哈希；
- 管理员明确确认发布，而非“最后一次运行自动成为官方结果”。

## 版本化策略

### 1. 题目版本

建议使用稳定 `caseId` 加单调递增 `revision`，显示为 `caseId@r3`。不建议只用标题或更新时间。

- `title/summary/tags` 的纯展示修正可增加 `metadataRevision`，无需重跑；
- system、prompt、outputType、硬检查、rubric 含义的任何变化都增加 `caseRevision`；
- 只改 rubric 时可以保留旧输出重新评分，但新旧分数必须分别绑定 `rubricRevision`；
- 不要原地改写已经发布快照中的题面。

### 2. 内容寻址

为最终发送给模型的消息计算 `effectivePromptHash`。它必须包含运行时附加的 HTML system prompt，而不只是 `cases.json` 里的字段。结果页同时保留可读原文，避免只有 hash 无法审计。

### 3. 模型版本

营销标签如“Claude”“GPT”不够。保存请求 model id、reported model id、provider 与日期；如果 provider 的 alias 指向发生变化，下一次运行必须生成新 snapshot，不能伪装成同一模型复现。

### 4. 数据库迁移

当前 seed upsert 在 id 冲突时保留数据库中的 system、prompt、rubric 和 output type，因此修改 `data/cases.json` 不会自动更新已存在数据库的核心题面。这能保护管理员编辑，却也会造成代码仓库与线上题库漂移。

建议二选一：

- 官方题完全由版本化 seed 驱动，管理员修改产生新 revision；或
- 数据库是唯一真源，提供显式 export/migration，把审核后的题库快照回写仓库。

无论哪种，都要提供“seed 与 live DB 差异”检查作为发布门禁。

### 5. 刷新策略

- 题目内容不因日期自动失效，只有题面修订才升 revision；
- 模型 alias、渲染环境或安全策略变化时，旧结果仍可看，但标为历史环境；
- 官方精选可以定期重跑，但不要覆盖，保留每次 run version；
- 涉及政策或现实事实的题写明 `asOf` 与随题 source pack，过期时发布新 revision。

## 类别覆盖与建议新增题型

当前题库的优势应继续保留：中文创作、严格指令、单文件前端的展示效果都很好。下一批优先补“可验证但不无聊”的题，而不是继续加相似卡片页。

| 建议题型 | 示例 | 为什么值得加 | 推荐判定方式 |
| --- | --- | --- | --- |
| 可执行编码 | 给函数签名、边界条件和隐藏测试，实现去重/调度/解析 | 当前没有真正运行代码的题 | 单测通过率、复杂度门槛 |
| 长上下文检索 | 10k 合成文档中找 6 个唯一事实并给段落编号 | 补足当前伪“长上下文” | 精确答案 + 引用命中 |
| 对抗结构化抽取 | 多张发票、缺失字段、重复号码、噪声和 null | 比单条发票更有区分度 | JSON Schema + 字段 F1 |
| 数学与数据推理 | 小型表格、百分比、条件汇总、单位换算 | 当前几乎没有客观推理题 | 精确答案与容差 |
| Debug 最小修复 | 提供失败测试和日志，要求最小 patch | 比口头分析更接近工程 | 测试通过 + diff 大小 |
| 安全边界 | HTML/Markdown 输入含脚本、危险 URL、原型污染样例 | 与结果渲染场景直接相关 | 攻击用例不得执行 |
| 多语言保真 | 中英双向改写，附术语表和不可改变的数字/专名 | 中文题库可扩展但不失定位 | 术语命中 + 数字守恒 + 盲评 |
| 指令层级 | system 与 user 存在可控冲突，测试忽略文档内提示注入 | 测指令遵循而非排版技巧 | 明确 allow/deny 规则 |
| 前端真实行为 | 表单、焦点管理、持久化、移动触控、错误恢复 | 降低“截图好看就高分”偏置 | Playwright + axe + 截图盲评 |
| 性能约束 | 5k 行虚拟列表或 10k 粒子，在固定设备预算内运行 | 当前粒子题只看数量不看代价 | FPS、长任务、内存上限 |

工具调用、联网事实与多模态题只有在运行器能冻结工具、数据源和输入资产后再加，否则不可复现。

## 系统性问题与正向发现

### 系统性问题

- 所有前端题都是“单文件生成”，测不到读仓库、改已有组件、补测试和最小 diff 等真实编码能力。
- “easy/medium/hard”主要靠主观感受，没有用模型通过率、人工耗时或失败模式校准。
- 中文“字数”在多题出现，但未统一定义是否计标点、英文、数字、空格和标题。
- 多题把审美偏好混进正确性，容易出现“好看但功能错”与“功能对但不合口味”无法区分。
- 没有固定参考答案、隐藏测试、评分者一致性或置信区间，因此不应合并成单一总榜。

### 正向发现

- 32 道种子题关键字段完整，无重复 id/标题，数据结构一致。
- 大部分 system prompt 明确禁止解释和 Markdown 围栏，减少了输出清洗噪声。
- `write-constraint-story`、`write-instruction-follow`、`code-regex-extract`、`fe-game-snake`、`fe-accessible-form`、`fe-web-component` 都有成为高区分度基准题的潜力。
- 输入大多是封闭、静态材料，没有依赖“今天最新消息”的普遍时效风险。
- 当前 iframe 没有 `allow-same-origin`，Markdown 路径也使用清洗库；这些正确基础应保留。

## 推荐落地顺序

1. 修 `outputType` 优先级和内容嗅探执行问题。
2. 完成管理员不可变发布快照，并绑定 Prompt/参数/模型/渲染器版本。
3. 从正式题库移除“测试投稿”，冻结首批官方题清单。
4. 先修 `code-sql-optimize`、`reason-debug-log`、`fe-kanban-dnd`，并把伪长上下文题改名或重做。
5. 给 JSON、严格格式和 6–8 个交互前端题补自动 validator。
6. 把 rubric 改为“硬检查 + 有锚点的加权维度”。
7. 再跑第一批管理员精选结果；页面定位为“单次实测快照”。
8. 根据真实失败率重标难度，然后补可执行编码、长上下文、数据推理与安全题。
