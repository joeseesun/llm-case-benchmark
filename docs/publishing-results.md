# 管理员题库结果发布手册

这套流程把“模型刚跑完的候选结果”和“访客默认看到的题库结果”分开。管理员手动跑一次并发布，可以作为该题的**单次实测快照**；它展示的是这一次运行，不代表稳定排名。

## 发布前准备

1. 打开 `/admin` 登录，在「模型」中确认本次要用的站点模型已启用；也可以使用当前浏览器里配置的模型。
2. 在「题库」中找到目标题，点击「运行 / 发布」。如果刚修改过题目的 Prompt、System Prompt 或输出类型，先保存题目，再重新运行。
3. 固定本次要比较的模型清单和参数。正式批次不要在模型跑到一半时临时增减列。

## 候选运行到正式发布

1. 选择模型，点击「运行并生成结果」。这一步只形成候选结果和普通运行历史，不会改变访客看到的精选版本。
2. 逐列检查：
   - 模型名称和预期清单一致；
   - 输出完整、非空、没有明显截断；
   - 错误列、延迟和用量信息符合实际；
   - 文本没有被错误当作 HTML 执行；
   - HTML/SVG 能正常打开，且没有不应公开的内容或未知外链。
3. 点击「发布为题库结果」，填写简短版本备注，例如“2026-07 首轮基准模型”。
4. 在确认框再次核对成功/失败数量，然后点击「确认发布」。发布完成后会生成 `v1`、`v2` 等新版本，并把新版本设为该题的精选结果。
5. 用退出登录的浏览器窗口打开题目，确认默认结果、模型列和版本备注都正确；再切换一次历史版本，确认旧结果仍能查看。

发布接口技术上允许保留失败列，只要求至少有一个成功结果。正式批次建议预定模型全部完成；如果确实要带失败列发布，应让失败状态原样显示，不要静默删掉该模型。

## 版本规则

- 每次确认发布都会新建版本，不会原地改写旧版本。
- 仅运行、单模型重跑或写入普通历史，都不会自动成为精选结果。
- 新发布版本会成为访客默认看到的结果，旧版本继续出现在历史版本中。
- 发布后再修改题目，不会回写旧快照；修改后的题目必须重新运行并发布新版本。
- 若误发了结果，当前做法是修正后再发布一个新版本。不要把再次运行误认为已经替换了线上精选结果。

## 安全边界

发布快照会保留公开复核所需的 Prompt、System Prompt、输出类型、模型/服务商信息、结果或错误、延迟、用量、评分、运行参数和版本备注。

快照**不会保存 API Key 或 Base URL**：

- 普通用户的 Key 只在该浏览器的 `localStorage` 中持久化；运行时会随请求临时经过服务端模型代理，但不会写入发布快照；
- 管理员配置的站点 Key 仍以加密形式保存在 SQLite；
- 公开结果接口只返回发布快照，不返回上述密钥或地址。

不要把 Key、内部地址、客户数据或其他秘密写进 Prompt、输出或版本备注；这些内容本身属于待发布内容，会随快照公开。

## 发布后验证

将 `CASE_ID` 替换成实际题目 ID：

```bash
# 公开题库只应包含已有精选结果的题目
curl -fsS 'https://benchmark.qiaomu.ai/api/cases?ready=only'

# featuredRun 是当前精选结果，history 是历史版本摘要
curl -fsS 'https://benchmark.qiaomu.ai/api/cases/CASE_ID/published-runs'

# 指定版本应能被独立读取
curl -fsS 'https://benchmark.qiaomu.ai/api/cases/CASE_ID/published-runs/1'
```

最后在桌面和手机宽度各检查一次页面。前端题额外确认预览无横向溢出、控制台致命错误或异常外部请求。
